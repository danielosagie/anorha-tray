# Ponder Tray-App Productionization — Recon + Plan

Goal: turn holo3-agent (Ponder/Anorha desktop) into a clean, signed, background **tray app** whose only job is: link account (Clerk) -> receive command -> execute over real browser+mouse -> wait for result -> pass result back -> surface success/error/offboarding. Multi-computer linking. Repos: holo3-agent, sssync-bknd, sssync_mobile_test.

Generated by recon workflow wghsqpfao (6 agents, 3 repos).

## Task lifecycle (end to end)

A command's end-to-end journey, hop by hop:

1) PRODUCER enqueues a job — EXISTS (partial). Mobile publish (sssync_mobile_test ProductDetail.tsx:2150 / PublishConfirmationScreen.tsx:76) POSTs /api/products/publish; backend's facebook.adapter.ts:110-129 enqueuePonderJob() calls Convex mutation browserJobs:create (convex/browserJobs.ts:115). MISSING for the tray vision: a generic, account-scoped "send this command to my computer(s)" producer. Today the only producer is the Facebook publish path; the Sprout agent has NO desktop-dispatch tool (confirmed — no publish_listing/dispatch tool in the mobile adapter, and getRetryable's ctx.auth is commented out).

2) JOB LANDS in the backend queue — EXISTS. Convex browserJobs table (sssync-bknd/convex/schema.ts:163-252) with full lifecycle (pending|processing|completed|failed), retry (attemptCount/maxAttempts/nextRetryAt/deadLetteredAt), velocity (queuePosition/nextEligibleAt/paused), runtime + executionPreference, and result/errorMessage. workerId field EXISTS on the doc but is NOT used to ROUTE a job to a specific machine.

3) BACKEND tells a desktop where to listen — EXISTS (weak). GET /api/agent/browser-jobs/bootstrap (agent.controller.ts:1399-1429, SupabaseAuthGuard) returns { convexURL, userId }. Desktop config.ts:164 bootstrapConfig() fills userId from a PONDER_BROWSER_JOBS_SYNC_TOKEN. MISSING: an actual account-LINK step (Clerk login on the desktop) — there is NO Clerk code in holo3-agent; identity is a userId string handed over a sync token, and any process holding a userId can poll.

4) LINKED DESKTOP claims the job — EXISTS (insecure). holo3-agent consumer.ts subscribes to Convex getRetryable({userId}) (browserJobs.ts:238), then startJob (browserJobs.ts:170). getRetryable has NO ctx.auth (line 112 is commented "In production: getUserIdentity") and takes userId as a plain arg → any worker knowing a userId drains that user's queue. No filter by target workerId. MISSING: per-device auth + targeted dispatch.

5) WORKER PRESENCE — EXISTS. workerPresence table (schema.ts:264-275), heartbeat upsert (workerPresence.ts:27-60), workerId derived from hostname (config.ts:99-108). Phone derives online = now-lastSeenAt < TTL.

6) EXECUTION on the real browser/mouse — EXISTS (strong, the moat). consumer.ts → ponder-executor.ts maps BrowserJob→Ponder action: recipe-first deterministic replay, fallback to vision agent_do over the Electron bridge (:7900, electron/main.ts:1632) → runAgentTaskForBridge (main.ts:557) → loop.ts observe-decide-act (screen.ts cliclick/nut-js for mouse+keyboard; playwriter.ts for Chrome CDP). Anti-loop guards, completion probe, decomposition all present. Friction/circuit-breaker for account safety (ponder-executor.ts:72-124).

7) WAIT for result — EXISTS (implicit, not contractual). MCP/bridge path blocks up to BRIDGE_RUN_TIMEOUT_MS=240s (mcp/tools.ts:425); the queue path is async (consumer awaits executor.execute then writes back). MISSING: an EXPLICIT "this blocks until done / here is the result shape" contract — the BridgeResult shape exists only in TypeScript.

8) RESULT passback — EXISTS. consumer.ts:379 completeJob / :452 failJob write result + errorMessage onto the Convex browserJobs doc; plus an HTTP reconcile callback (consumer.ts:463 → /api/agent/sessions/:id/browser-jobs/:id/reconcile). Ponder also persists its own session/steps in its LOCAL Convex (holo3-agent/convex/schema.ts).

9) RESULT surfaced to the user — EXISTS on mobile, MISSING in the tray. Mobile useFacebookJobStatus.ts:72-140 maps job state to dot+label (Posting…, Queued, Live, Couldn't post, Needs a check, offline-waiting) realtime via the browserJobs Convex client; listingUrl shown on completed. The TRAY app itself has no clean success/error surface tied to the queue — it currently shows an in-app Run page + Buddy bubble driven by MCP, not by the browserJobs queue result.

NET: hops 2, 5, 6, 8 are solid. Hops 1 (generic producer), 3/4 (Clerk account-link + secure, per-device targeted claim), 7 (explicit result contract), and 9 (tray-native result surface) are the build frontier.

## KEEP (works, tray-essential — do not rebuild)
- Convex browserJobs queue (sssync-bknd/convex/schema.ts:163-252 + browserJobs.ts create/startJob/completeJob/failJob/getRetryable/getForUser) — full lifecycle, retry, velocity, result fields. This is the spine; do not rebuild.
- workerPresence heartbeat (sssync-bknd/convex/workerPresence.ts + holo3-agent consumer heartbeat, hostname workerId) — the 'computer online' signal the tray and phone both need.
- holo3-agent execution engine: loop.ts (observe-decide-act), brain.ts, screen.ts (cliclick/nut-js mouse+keyboard), playwriter.ts (Chrome CDP), eyes.ts grounding, anti-loop guards, completion probe, decompose — the actual 'mastery of browser + mouse'.
- Recipe record/replay (ponder-executor recipe-first, ~/.ponder/recipes, ponder_recipe_* / ponder_session_replay) — deterministic fast re-execution is the performance moat; keep as the primary path with vision fallback.
- Electron HTTP bridge :7900 (electron/main.ts:837-2355): /health, /agent_do, /browser/*, /screen/*, /recipe/* with Bearer-token auth (src/bridge/auth.ts). Keep — it's how a job actually drives the machine.
- Account-safety: friction-phrase detection + circuit breaker + write velocity caps (ponder-executor.ts:72-124, consumer.ts) — essential for not getting marketplace accounts banned.
- Tray icon + menu infrastructure (electron/main.ts:3097-3222 buildTray/rebuildTrayMenu) and the Buddy overlay window (windows.ts:189-231) — tray-first foundation already exists.
- macOS permission probes (src/perms.ts: Accessibility, Screen Recording, Input Monitoring) — required for onboarding and error recovery.
- Channel sign-in (openChannel IPC → Facebook/eBay/Poshmark/Mercari/Google in default browser) — user must sign in once; keep, but move into the clean onboarding.
- Result passback: completeJob/failJob writing result+errorMessage + reconcile callback (consumer.ts:379/452/463) — the return path works.
- Mobile job-status realtime UX vocabulary (useFacebookJobStatus.ts:72-140 dot+label states, BrowserJobsConvexProvider, watchQuery not useQuery) — reuse the same state model in the tray's result surface.
- Bootstrap endpoint GET /api/agent/browser-jobs/bootstrap (agent.controller.ts:1399-1429) — keep as the convexURL/userId discovery channel, but harden the identity it returns.

- Hero composer free-text 'what do you want done?' task input in src/renderer/app/App.tsx:248-299 (RunIdle) — this is the tinkerer entrypoint; the tray RECEIVES commands from the linked account, it is not a prompt box. Strip or hide behind a dev flag.

- Dead routing/planner paths now bypassed in composite mode: createOllamaPlanner / makeRouter / src/agent/planner.ts hierarchical planner (loop.ts:673-689 already passes router:null) — remove to shrink the binary and reduce confusion.
- Ollama narrator (createOllamaNarrator, electron/main.ts:284) — the 'hi i'm holo3' speech intro; cosmetic tinkerer flourish, zero loop integration.
- AGP subsystem (runAgpTask / AgpClient in electron/main.ts + loop.ts) IF the tray is desktop-browser-only — verify it's unused first; it's a separate mobile-automation brain not needed for the linked-computer execution vision.
- Mobile Expo push userDevices table (sssync-bknd/src/db/schema.ts:35-45) is NOT desktop linking — do not conflate it with device-linking; leave it alone (it's for phone push), just don't build desktop linking on top of it.

## BUILD (dependency order)
1. Clerk account-link on the desktop (holo3-agent) — there is ZERO Clerk code today. Add a Clerk sign-in in the Electron app so a machine is bound to a real account, mint a per-device secret, and stop relying on a bare userId string. Dependency for everything below.
2. Per-device identity + secure job claim — add a deviceId/deviceSecret to workerPresence and ENFORCE ctx.auth (or device-secret verification) on getRetryable/startJob in sssync-bknd/convex/browserJobs.ts (currently auth is commented out at line 112; any userId can drain the queue). Security-critical.
3. Targeted multi-computer dispatch — add a targetWorkerId to browserJobs:create + filter getRetryable by it, so the phone/account can send a job to 'Office Mac' vs 'Home PC'. The workerId column exists but is not used for routing.
4. Tray-native result surface — a clean success/error panel in the Electron app driven by the browserJobs queue result (reuse the mobile state vocabulary from useFacebookJobStatus.ts), including the 'something went wrong — turn this on' recovery state. The tray currently surfaces MCP runs, not queue results.
5. Generic 'execute this command' producer — beyond Facebook publish. Either a Sprout-agent dispatch tool or a generic command enqueue so the account can send arbitrary commands to a linked machine (today only facebook.adapter enqueues).
6. Explicit result contract — formalize the BridgeResult/job-result shape (outcome enum, result, errorMessage, optional artifacts/screenshot) and document the wait semantics, so the producer/UI can render success/error deterministically.
7. electron-builder (or forge) packaging — NONE exists. Add appId/productName, icon, Info.plist, entitlements (hardened runtime + Accessibility/Screen Recording usage strings).
8. Code signing + notarization — Developer ID Application signing + notarytool stapling in the build pipeline (zero signing config today). Required for a downloadable installer that opens without Gatekeeper blocking.
9. DMG installer + auto-update — electron-updater with a signed update feed so linked machines stay current (no update mechanism exists).
10. Clean onboarding flow — link account (Clerk) → grant permissions (perms.ts) → sign in channels → name this computer → confirm reachable. Tie LinkComputerSheet's reachability probe to a real linked state.
11. Real 'open Ponder' deep link — the ponder:// scheme is only a canOpenURL probe today (LinkComputerSheet.tsx:36); GET_APP_URL just opens a web download page. Implement the scheme so the phone's 'turn this on' actually focuses/launches the app.
12. Offboarding/unlink — a way to unlink a device from the account (revoke deviceSecret, clear workerPresence) from both the tray Settings and the mobile Connections screen (mobile list is read-only today).

## Hardest gaps
- **No account identity on the desktop — holo3-agent has no Clerk integration; it trusts a userId handed over a sync token.**
  - impact: The core product promise ('link the account') is unmet, and the queue is unauthenticated (getRetryable has ctx.auth commented out at browserJobs.ts:112) so any userId drains another user's jobs.
  - rec: Add Clerk sign-in in the Electron app; on login register a device (deviceId+deviceSecret) via a new backend endpoint; enforce auth on getRetryable/startJob/completeJob/failJob. Make this the very first build phase — multi-computer + security both depend on it.
- **Jobs cannot target a specific linked machine.**
  - impact: User's explicit requirement (link MULTIPLE computers, dispatch to one) is impossible; today any online worker for that userId claims the job.
  - rec: Add targetWorkerId to browserJobs:create and filter getRetryable by (userId, targetWorkerId or null); expose a machine picker in the mobile dispatch UI and tray. Build right after device identity.
- **Zero packaging/signing/notarization infrastructure (no electron-builder, no entitlements, no notarytool, no DMG, no auto-update).**
  - impact: There is no downloadable, openable, trusted installer — a hard blocker for distribution; an unsigned Electron app needing Accessibility + Screen Recording will be blocked by Gatekeeper.
  - rec: Adopt electron-builder with Developer ID signing + notarytool stapling + hardened-runtime entitlements + usage strings, plus electron-updater. Reuse the user's existing Apple Developer account/keys (same identity used for the anorha iOS releases).
- **Result wait + return is implicit, not contractual.**
  - impact: The producer/UI can't reliably know when a job finished or render a clean success/error; the 240s block is buried in a constant.
  - rec: Define a typed JobResult { outcome, result, errorMessage, artifacts?, screenshot? } persisted on the Convex doc and returned by the bridge; the tray + mobile both subscribe to the doc (realtime) rather than blocking. Treat the queue doc as the single source of truth.
- **Tray app surfaces MCP/tinkerer runs, not the linked-account queue results.**
  - impact: The user-facing success/error states the vision demands ('something went wrong, turn this on') don't exist in the tray; they only exist on mobile.
  - rec: Build a tray result panel that subscribes to the browserJobs queue (reusing useFacebookJobStatus state vocabulary) and strip the hero composer/chat. Tray = status + recovery, not authoring.
- **The ponder:// deep link is a probe only; 'open Ponder' actually opens a web download page.**
  - impact: The phone's 'turn your computer on' recovery CTA can't actually wake the app.
  - rec: Register a real ponder:// (or anorha-desktop://) URL scheme + LSApplicationQueriesSchemes so the phone can focus/launch the running tray app, and so deep-link onboarding works.
- **No device management surface (rename/unlink) on either side.**
  - impact: Multi-computer is unusable in practice — user can't tell machines apart or revoke a lost laptop; offboarding is undefined.
  - rec: Add device naming at link time and an unlink/revoke action in both tray Settings and mobile Connections (currently read-only).

## Design surfaces (drives Paper phase)
- **Tray menu (menu-bar dropdown)** — The always-present home of the app; quick glance at status and the few real controls.
  - states: Linked + idle (green dot, account email, 'Ready'), Running a job (spinner + current job title), Not linked (only 'Sign in' visible), Error/needs-attention (amber dot, 'Action needed'), Offline/paused (gray dot), Quit / Open settings
- **Onboarding — Welcome + Sign in (Clerk)** — Link this machine to the account. First and mandatory step.
  - states: Welcome/value (one line), Clerk sign-in (email/OAuth), Signing in (spinner), Sign-in error
- **Onboarding — Grant permissions** — Get Accessibility + Screen Recording + Input Monitoring so the agent can drive the real browser/mouse.
  - states: Checklist (each perm: not granted / granted), Requesting (system prompt up), All granted (advance), Blocked (deep link to System Settings + Re-check)
- **Onboarding — Sign in to channels** — One-time marketplace logins (Facebook/eBay/etc.) in the real browser.
  - states: Channel list (signed-in / not), Opening browser, Confirmed signed-in, Skip for now
- **Onboarding — Name this computer** — Give the machine an identity for multi-computer targeting.
  - states: Name entry (default = hostname), Confirming/registering device, Linked success (handoff to idle)
- **Idle / Ready (main panel when opened)** — Calm 'everything's good, waiting for commands' state.
  - states: Linked + reachable + idle, Last result card (trivial history: title + outcome + open-listing link), Velocity-paused ('catching up')
- **Running / Live** — Show that a command from the account is executing right now.
  - states: Receiving job, Executing (live step line / Buddy bubble), Recipe replay vs vision (subtle), Long-running (progress), Cancel in flight
- **Result — Success** — Cleanly confirm a command finished and surface the artifact.
  - states: Done (checkmark + summary), Done with listing/result link, Done, queued next
- **Error / Recovery** — The 'something went wrong — turn this on' surface; the single most important state.
  - states: Permission missing → fix CTA, Channel logged out → re-sign CTA, Friction/circuit-breaker tripped → reset CTA, Browser/Chrome not reachable, Job failed (retry / dismiss), Computer was offline (will run when on)
- **Offline / Disconnected** — Communicate the machine can't currently take jobs.
  - states: Account session expired (re-link), No network, Backend unreachable, App backgrounded but queue waiting
- **Settings** — The minimal real settings for a tray app.
  - states: Account (email, sign out), This computer (name, deviceId, unlink), Permissions status, Channels status, Launch at login + updates, Danger zone (unlink/revoke)
- **Offboarding / Unlink** — Safely remove this machine from the account.
  - states: Confirm unlink, Revoking, Unlinked (back to sign-in)
- **User journey map (cross-surface)** — End-to-end story to validate the flow before building.
  - states: Download → install (signed DMG), First open → Clerk link, Grant perms, Sign in channels, Name computer → Linked, Phone/account sends a command, Tray receives → executes (browser/mouse), Result returns → surfaced on tray + phone, Error path → 'turn this on' recovery, Add a 2nd computer → target a specific one, Unlink a computer

## Recommended sequence
1. PHASE 0 — Design in Paper (no code): build 2-3 versions of the full tray surface set above + the user journey map, in the anorha design language (minimal words, no emojis, Icon components). Validate idle/running/result/error/offline + onboarding/offboarding before any build. This phase directly consumes designSurfaces.
2. PHASE 1 — Identity + security (sssync-bknd + holo3-agent): add Clerk sign-in to the Electron app; add a device-register endpoint minting deviceId+deviceSecret; ENFORCE auth on convex/browserJobs.ts getRetryable/startJob/completeJob/failJob (uncomment/replace the ctx.auth at line 112); add deviceSecret to workerPresence. Unblocks everything.
3. PHASE 2 — Targeted multi-computer dispatch (sssync-bknd): add targetWorkerId to browserJobs:create and filter getRetryable; expose machine identity in workerPresence; add a generic command-enqueue producer (not just Facebook).
4. PHASE 3 — Tray UX rebuild (holo3-agent renderer/electron): strip the hero composer, Overlay chat, provider/engine chooser, narrator, auto-show window, dead planner/router paths; build the tray-native result surface subscribing to the browserJobs queue (reuse useFacebookJobStatus state vocabulary); wire idle/running/result/error/offline; implement onboarding (link→perms→channels→name) and offboarding (unlink).
5. PHASE 4 — Result contract (holo3-agent + sssync-bknd): formalize the typed JobResult shape on the bridge + Convex doc and document wait semantics; ensure both tray and mobile render success/error from the doc, not from blocking calls.
6. PHASE 5 — Deep link + reachability (holo3-agent + sssync_mobile_test): register the real ponder:// URL scheme so the phone's 'turn this on' launches/focuses the app; replace LinkComputerSheet's web-download fallback with the live deep link; tie reachability probe to true linked state.
7. PHASE 6 — Packaging + signing + updates (holo3-agent): add electron-builder (appId, icon, Info.plist, hardened-runtime entitlements + Accessibility/Screen-Recording usage strings), Developer ID signing, notarytool stapling, DMG, and electron-updater feed. Produce a downloadable installer.
8. PHASE 7 — Device management polish (both): rename/unlink in tray Settings + mobile Connections (today read-only); confirm offboarding revokes deviceSecret + clears presence.
9. Then loop: device-test the full link→dispatch→execute→result→error cycle across two linked machines; iterate on the Paper designs where reality diverges.

## Open questions
- Auth provider on the desktop: confirm Clerk (matches mobile + backend convex/auth.config.ts) is the intended desktop identity, and whether desktop should use Clerk OAuth/device-code flow vs email. There is currently NO Clerk code in holo3-agent.
- Apple signing identity: which Developer ID Application cert + notarytool credentials should the Electron build use — the same Apple Developer account used for the anorha iOS releases? Notarization needs an app-specific password or ASC API key.
- Keep task history at all? User said 'only if trivially retained' — confirm whether the tray shows just the last result card or a small recent-results list (drives whether the History tab/component survives).
- Targeted dispatch model: when multiple computers are online, should the account/phone always pick a target machine, or default to 'any available' with optional pinning? This shapes the browserJobs:create API change.
- Is the AGP subsystem (runAgpTask/AgpClient) used by any current flow, or safe to strip for a desktop-browser-only tray? Need a usage confirmation before deleting.
- Provider pinning: which single execution provider should the shipped tray use (hcompany AGP vs composite Gemini+Holo3 vs Modal) now that the chooser is being removed? Memory notes preferences.json pins provider over env.
- Cross-platform: is the installer macOS-only for v1, or does Windows (the user's secondary GTX-1650 PC) need packaging too? cliclick/nut-js and screencapture are macOS-specific.
- Should the desktop consumer run inside the Electron main process (single app) or as a separate 'ponder consume' process? Today both paths exist; pick one for the tray to keep lifecycle simple.
