## Clerk plan
"Follow the approach already prescribed in ponder-tray/docs/PHASE1-IDENTITY.md (the repo's own decision): Clerk is used ONCE at link time, NOT kept live in the background. Least-risky Electron approach = run the OAuth flow in the SYSTEM browser (BrowserWindow OAuth inside Electron is fragile and breaks Google/Facebook 'disallowed_useragent' checks). Recommended: use @clerk/clerk-js (headless) in the renderer with a deep-link/loopback callback, OR a Clerk device-authorization/ticket flow handled in main, ending with a Clerk session token. Package: add @clerk/clerk-js (the lockfile already references @clerk/clerk-react/@clerk/react but nothing is installed). Publishable key: NOT present in ponder-tray — surface it via env (CLERK_PUBLISHABLE_KEY) and return it through the extended env:public handler; the prod value used by mobile is pk_live_Y2xlcmsuYXBwLmFub3JoYS5hcHAk (clerk.app.anorha.app) per sssync_mobile_test/.env.local:33 — must use the SAME prod instance (clerk.app.anorha.app) so the token validates against the backend Convex auth.config (issuer https://clerk.app.anorha.app). Session token: call getToken() with NO template (mobile uses the plain session token, sub=Clerk user id — ConvexProvider.tsx:46-79; there is no 'convex' template). orgId: resolve the same way mobile does — GET ${API_BASE}/api/organizations/me/active with Authorization: Bearer <clerkToken> (OrgContext.tsx:155-189) to get the active org's id; registerDevice requires orgId as an explicit arg (it is NOT auto-detected). Token handoff to main: the renderer obtains the Clerk token + orgId, passes both to main via window.agent.registerDevice; main calls the exported registerDevice({convexURL: VITE_CONVEX_URL, clerkToken, orgId, name, platform}) which writes device.json. convexURL for the QUEUE must be the browserJobs deployment https://merry-buffalo-800.convex.cloud (sssync-bknd/.env:37), NOT the mobile agent-chat deployment trustworthy-porpoise-787 — confirm which VITE_CONVEX_URL the tray ships with. Electron caveat: do NOT attempt OAuth inside an in-app BrowserWindow webview; use the system browser + a callback (custom scheme / loopback) to avoid provider user-agent blocks."
