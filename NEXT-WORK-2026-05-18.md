## Item 1 — Native-resolution crop (SPEED). Do this first.

### Problem
`maybeCropToTargetApp` (`src/agent/loop.ts:262`) crops the target
window at **logical resolution** (`scaleFactor=1`): for Calculator
it emits a 230×408 ≈ 8 KB thumbnail (dev log: `→8447 bytes, ~15.8×
fewer pixels`). The model grounds that degraded image wrong ~100% of
the time (observed: "4" → (114,399)). So the `MIN_CROP_DIM_PX = 300`
guard (`loop.ts:~334`) correctly forces sub-300px windows to run
**uncropped** — i.e. the full 1512×982 / ~1.2 MB frame every step,
which is the ~10s/call Modal bottleneck.

`bench/vision-precision.ts` proves the fix: its `cropPng` crops the
**native/Retina PNG** (460×816 for the same window) and grounds at
**8/8, ~5px**. Native-res crop is both accurate AND far fewer pixels
than full-screen → the real speedup.

### Fix
Make `maybeCropToTargetApp` crop from the **native-resolution**
screenshot, not a logical-downsampled one. Mirror the proven math in
`bench/vision-precision.ts` (~lines 300–345):

1. Read PNG IHDR dims from `shot.png`; `scaleX = pngW / shot.width`,
   `scaleY = pngH / shot.height` (2 on Retina).
2. Crop rect in **PNG-pixel space**:
   `x=(bounds.x-shot.offsetX)*scaleX`, `y=…*scaleY`,
   `w=bounds.width*scaleX`, `h=bounds.height*scaleY`.
3. Produce the crop **without downscaling** (keep native px). If
   `nativeImage` is downsampling, switch to the bench's `sips`
   approach (`src/agent/imageops.ts cropAndScalePng` with scale=1
   already does an exact native crop — reuse it).
4. Keep `screen=[bounds.width, bounds.height]` (logical) as the
   grounding coord space; the loop's existing offset translation
   (`coords + shot.offsetX/Y`, plus `+bounds.x/y` for the crop
   origin) maps back. Verify the translation matches the bench's
   `r.x + bounds.x` exactly.
5. Only AFTER native-res crop is confirmed: lower/remove
   `MIN_CROP_DIM_PX` (env: `PONDER_MIN_CROP_DIM_PX`). Keep a tiny
   sanity floor (~80px) + the existing multi-monitor recapture and
   off-screen-fit checks (orthogonal, keep them).

### Validation (offline, fast — no 20s/step loop)

- One `calculator-mouse-math` run: dev log must show

### Scope / risk

## Item 2 — Brain decomposition / sequencing (STABILITY).

### Problem
Flat `agent_do` skips the planner (`runTask` flat branch,
`loop.ts:~604`; the Ollama planner was disabled because it
over-decomposed — rationale at `loop.ts:~586`). Holo3 is
grounding-first: it executes single clicks accurately but (a) never
emits DONE on a multi-step task, and (b) mis-sequences (observed:
double-entered "4", looped `8,=,8,=`, never reached 47×8). The
committed completion probe mitigates (a); (b) is unsolved and is the
real "make the app stable" problem.

### Root cause
No task-state tracking. Each step the brain re-plans from the
screenshot + a flat history list; it has no notion of "which
sub-step am I on / what did I already do / am I done." Holo3 is not
a planner and prompt-engineering termination into the small model is
documented as fragile (`brain.ts` preamble comments).

### Decision

**Option A (one-shot strong-model decomposition + verify-to-advance),
paired with C (deterministic done-signals where cheap).** Rejected B
(structured prompt memory) — the small model provably drops structure
(`brain.ts` preamble comments). A is *not* the disabled Ollama
planner: that decomposed every flat call into wrong sub-tasks with a
weak local model; this is ONE call to the strong hosted model, only
for declared-multi-step tasks, with explicit position tracking and a
verifier gate between items.

### Implementation-ready spec

**New module `src/agent/decompose.ts`:**
```ts
export interface Plan { steps: string[]; oneShot: boolean }
// decompose(task, firstScreenshotB64, provider, signal): Promise<Plan>
//  - ONE provider.plan() call with a strict prompt: "Return a JSON
//    array of atomic UI steps to accomplish: <task>. Each step = one
//    physical action a clicker can verify is done from a screenshot.
//    If it's already a single atomic action, return [task]."
//  - Parse JSON; on any parse failure or len<=1 → { steps:[task],
//    oneShot:true } (degrade to today's flat behavior — never worse).
//  - Hard cap N steps (e.g. 12); longer → oneShot fallback + log.
```

**Loop integration (`src/agent/loop.ts`):**
- Gate: `PONDER_DECOMPOSE=on` AND task is multi-step. Detection:
  the bench passes it (case frontmatter `tier` ∈ {T3,T4,T5} or a new
  `multistep: true`); the bridge passes an explicit `decompose:true`
  in the `/agent_do` body. Default OFF → zero behavior change.
- In `runTask` flat branch (`loop.ts:~604`, where it currently does
  `runOneSubtask({...,task})`): if gated on, call `decompose()` once,
  then loop the steps — each step is its own `runOneSubtask` call
  with `task = steps[i]`, `overallGoal = <original task>`, and a
  small per-step `maxSteps` (e.g. 8).
- **Advance contract:** a step is "done" when `runOneSubtask` returns
  `"done"` OR the existing completion probe / `verify()` confirms
  that step's sub-goal. Only then `i++`. If a step returns
  `"exhausted"` twice, abort the whole task `exhausted` (don't grind
  later steps against a broken precondition). `infeasible` from any
  step → propagate (existing logic).
- **C pairing:** before/after each step, if a cheap deterministic
  signal exists for the surface, use it instead of a model verify
  (e.g. Calculator: read display via the AX/`System Events` path
  already used by `bench`/`screen.ts`; "expression entered" / "result
  shown" are exact). Keep this a small optional `domainSignals`
  table, not a generalization attempt.

**Reuse (no new infra):** `verifier.ts verify()` for advance gating;
the committed completion probe for intra-step termination; the
existing `TaskOutcome` union + subtask aggregator for outcomes.

### Validation (offline-first, after Item 1 lands)
1. `calculator-mouse-math` via the bridge: with `decompose:on` the
   dev log should show a 5-step plan (4,7,×,8,=), each advancing on a
   verified display state, terminating DONE with `376` shown — no
   `8,=,8,=` loop. Per-step ~3–5s (Item 1 speed).
2. One T4 case via `bench/run-all.ts`; compare outcome vs flat.
3. Confirm `decompose:off` is byte-identical to today (flat path).

### Scope / estimate
~150–250 LOC (`decompose.ts` + the flat-branch loop). One focused
session AFTER Item 1 is validated. Do NOT implement before Item 1 —
without ~3–5s/step it is not measurable and regressions hide.
